# 🧠 에이전트 개선의 7가지 교훈

멀티 에이전트 파이프라인(수집 → 분석 → 시각화)을 반복 개선하며 얻은 핵심 교훈입니다.  
각 교훈에는 **관찰**(문제 발견) → **해결**(적용한 원칙) → **시사점**(일반화된 교훈)을 담았습니다.

---

## 💡 Lesson 1: 에이전트는 '실행기'가 아닌 '전략가'여야 한다

* **관찰**: "페이지에서 데이터를 수집해"라고만 하면, 에이전트는 'Next' 버튼만 무한히 클릭합니다.
* **해결**: DOM 탐색 전에 **전략 풀(Strategy Pool)**을 프롬프트로 내재화했습니다. (예: URL 패턴 분석, 에스컬레이션 사다리)
* **시사점**: 에이전트 간 메시지에는 행동 지침뿐 아니라 **"어떤 전략을 채택했고 왜인지"**라는 메타 인지가 포함되어야 합니다.

## 💡 Lesson 2: 도구가 곧 에이전트의 지능이다

* **관찰**: 프롬프트를 아무리 정교하게 써도 행동의 한계가 있었습니다.
* **해결**: Navigator에게 `get_page_structure`를 주어 분석가로, Coder에게 `run_command`를 주어 개발자로 만들었습니다.
* **시사점**: **프롬프트는 '성격'을, 도구는 '능력'을 결정합니다.** 도구의 입출력과 Docstring 설계가 성능의 핵심입니다.

## 💡 Lesson 3: 프롬프트 엔지니어링 → 컨텍스트 엔지니어링

* **관찰**: 시스템 프롬프트에 모든 예외를 넣자 토큰이 낭비되고 모델의 주의력이 분산되었습니다.
* **해결**: **런타임**과 **미들웨어(Middleware)**로 필요한 순간에만 정보를 주입했습니다.
* **시사점**: "무엇을 쓸까"보다 어떻게 **"필요한 정보를 적절한 시점에 적절한 양만큼 전달할까"**가 컨텍스트 엔지니어링의 핵심입니다.

## 💡 Lesson 4: 에러를 통한 지속적 학습

* **관찰**: 모든 사이트의 방어 기제를 사전에 프롬프트에 담는 것은 불가능합니다.
* **해결**: 실패 경험을 **패턴 메모리(Pattern Memory)**에 저장하고, 다음 작업에 RAG로 주입했습니다.
* **시사점**: 한 번 겪은 에러와 해결책을 시스템의 자산으로 축적하는 구조를 만드세요.

## 💡 Lesson 5: 평가 주도 개발 (Evaluation-Driven Development)

* **관찰**: "잘 되는 것 같다"는 직관만으로는 변경의 영향을 파악할 수 없습니다.
* **해결**: 시나리오 + 평가 기준 + LLM-as-a-Judge를 결합한 **테스트 하네스**를 구축했습니다.
* **시사점**: 에이전트의 결과는 정답이 아니라 '범위(Spectrum)'입니다. TDD → **EDD**로 진화하세요.

## 💡 Lesson 6: 결정론(코드) × 비결정론(LLM) — 하이브리드 인터페이스

* **관찰**: 에이전트 시스템은 본질적으로 **"두 개의 뇌"**가 협력해야 합니다.
  - **파이썬 코드**(결정론): 정확한 Key와 데이터 타입을 요구합니다.
  - **LLM**(비결정론): 의도와 맥락 같은 풍부한 자연어를 요구합니다.

  이 둘의 요구사항을 분리해서 관리하면(예: 지시는 `.txt`, 채점 기준은 `.json`) **동기화가 어긋나** 파이프라인이 쉽게 붕괴됩니다. 지시 파일에서 필드명을 바꿨는데 JSON 파일은 안 바꾸는 식의 실수가 반복됩니다.

* **해결**: 파일 하나에 **기계용 규격**과 **LLM용 지시**를 결합한 하이브리드 문서 구조를 채택했습니다.
  ```markdown
  ---
  # YAML Frontmatter (파이썬이 파싱하는 엄격한 규격)
  required_fields: [title, price, rating]
  min_items: 20
  scoring_weights: {completeness: 0.4, accuracy: 0.6}
  ---
  # Markdown (LLM이 읽는 자연어 지시)
  네이버 쇼핑에서 "무선 키보드"를 검색하고 상위 20개 상품의
  제목, 가격, 평점을 수집하세요. 광고 상품은 제외합니다.
  ```
  이렇게 하면 **"무엇을 수집할지"와 "어떻게 채점할지"가 항상 같은 파일에서 동기화**됩니다.

* **시사점**: 통제 불가능해 보이는 AI를 기존 소프트웨어 공학의 통제권 안으로 끌어들이려면, 기계와 LLM이 완벽히 동기화된 **'단일 진실 공급원(Single Source of Truth)'**이 필요합니다. 명령과 규격을 분리하지 말고 **하나의 원자(Atom) 단위**로 결합하세요.

## 💡 Lesson 7: 시스템 프롬프트 엔지니어링 — 모듈화, 원칙, 계층 분리

프롬프트를 개선하며 발견한 세 가지 핵심 원칙입니다.

### 7-1. 프롬프트가 곧 코드다 (Prompt-as-Code)

* **관찰**: 워크플로우 코드 안에 프롬프트를 하드코딩하면 수정과 버전 관리가 고통입니다.
* **해결**: 프롬프트를 `app/prompts/` 패키지로 독립시키고 모듈화했습니다.
* **원칙**: 프롬프트도 코드와 동일하게 **모듈화하고 버전 관리**하세요.

### 7-2. 프롬프트는 '패치'가 아니라 '원칙'이다 (Generalization over Overfitting)

* **관찰**: "25개 기대, 13개만 수집되면 정상으로 인정하라"와 같은 종종 예외 규칙을 프롬프트에 기술하곤 합니다. 이것은 하나의 사례에 대한 과적합된 패치입니다.
* **해결**: 특정 케이스에 대한 규칙을 만들지 말고 **범용적 의사결정 원칙**으로 대체해야 합니다.

> **과적합 버전 ❌**
> ```
> 데이터 수집 개수 부족(예: 25개 기대, 13개 수집)과 같은 검증 오류의 경우,
> 대상 페이지의 실제 데이터 양이 적어서 발생하는 문제일 수 있습니다.
> 이 때는 무의미한 코드 수정을 멈추고 즉시 작업을 중단하세요.
> ```
> 📌 숫자가 사례를 그대로 반영. 원인을 "데이터 부족"으로 단정하여, 진짜 버그에도 포기하게 만듦.

> **범용 원칙 버전 ✅**
> ```
> 에러나 검증 실패가 발생하면, 재시도 전에 반드시 원인을 분류하세요:
>   a) 내 코드의 버그인가? → 코드를 수정하고 재시도. (최대 3회)
>   b) 코드 외부의 요인인가? → 코드 수정은 무의미하므로 즉시 중단하고 보고.
> ```
> 📌 "원인 분류"라는 사고 프레임워크를 제공하여 어떤 실패에도 적용 가능.

* **원칙**: 버그를 고칠 때 스스로에게 물으세요 — *"이 수정은 이 한 건만 고치는 패치인가, 유사한 모든 상황을 커버하는 원칙인가?"*

### 7-3. 정보의 3계층 분리 (Separation of Concerns)

* **관찰**: 시스템 프롬프트에 도구 파라미터명, 특정 사이트 예시, 권장 라이브러리까지 전부 넣자, 프롬프트가 매뉴얼처럼 변질되어 AI가 **'지시 수용 모드'에 갇혔습니다.**
* **해결**: 정보의 성격에 따라 3계층으로 분리했습니다.

| 계층 | 내용 | 위치 |
|:---|:---|:---|
| **Layer 1**: Identity & Principles | "당신은 전략가다", "에러 시 원인 분류하라" | 시스템 프롬프트 |
| **Layer 2**: Tool Contracts | `mode="chat"`, `wait_seconds=5` 등 구현 디테일 | 도구 Docstring |
| **Layer 3**: Task Context | 특정 사이트 특성, 이번 작업 제약 | 런타임 주입 |

* **원칙**: **프롬프트에는 'What(무엇을)'과 'Why(왜)'만 남기고, 'How(어떻게)'는 도구 명세에 양보하세요.** 프롬프트 다이어트는 비용 절감이자 지능 향상에 중요합니다.
